5. 风险评估矩阵 (含峰值压力预测)

#风险项等级描述缓解措施
1核心应用峰值内存激增 高 ① 地图导航：在高精地图渲染/复杂路况下峰值可达 1GB。
② 语音 AI：当前已占用 1.8GB，持续交互可能触及更高上限。
③ 360环视/AVM：开启全景影像时多路摄像头拼接峰值约 1GB。
④ system_server：随系统运行时间增加，常驻内存可能涨至 1GB。		 ① 限制 GVM 低优先级缓存，确保核心进程有 4GB+ 弹性余量
② 启用更加激进的 LMKD 策略
③ 优化地图 3D 纹理与语音模型加载策略
2GPU/DMA-BUF 峰值不足 中 当前 DMA-BUF 已分配 6GB(含 unmapped pool 1.3G)。多屏同时渲染高精地图+3D车模+360环视时，可能触及峰值上限。 ① 实测全场景峰值 DMA-BUF 使用量
② 评估纹理压缩/降分辨率方案
③ 设置 DMA-BUF 上限预警
3PVM Buffer 映射风险 中 PVM 9.3GB buffers 中包含 GPU 后端存储。在 48GB 方案中，若 PVM buffer 被过度压缩，可能导致渲染掉帧。 ① 建立 PVM Buffer 动态监控预警
② 预留足够的 PVM Slab 内存
4多用户并行压力 中 副驾 User 10 并行运行娱乐应用。与主驾高负载场景叠加时，总 PSS 可能突破 25GB。 ① 对非关键 User 进程设置更严格的内存限制
② 限制后台应用驻留数量

6. 降本实施方案建议

推荐策略：分阶段验证

✅ 总体结论：64GB → 48GB 降本在技术上是可行的，当前系统存在约 29GB 的综合空闲空间， 即使保守削减 16GB，仍有充足余量。但需要通过分阶段测试验证 GPU/DMA-BUF 峰值和 PVM Buffer 的最小需求。

Phase 1 — 软件层面验证（无需改硬件）

步骤操作验证指标
1.1通过 kernel cmdline 将 GVM 内存分配从 33GB 降至 23GB（mem=参数）GVM 能否正常启动、所有应用是否正常运行
1.2全场景压力测试：导航+音乐+语音+ADAS+副驾视频 同时运行dumpsys meminfo 的 Used RAM、DMA-BUF 峰值、是否触发 OOM
1.3GPU 渲染压力测试：多屏同时渲染复杂 3D 场景GPU total allocated、DMA-BUF mapped、帧率是否下降
1.472 小时长稳测试，监测内存泄漏内存增长趋势、是否出现 ZRAM swap 激增

Phase 2 — 硬件验证（确认可行后）

步骤操作验证指标
2.1更换 48GB 样机，烧录适配后的固件系统正常启动、所有功能可用
2.2完整 QA 测试套件功能回归、性能回归、稳定性回归
2.3温升和功耗测试降本不增加热风险
