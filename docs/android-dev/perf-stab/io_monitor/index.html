IO 监控参数小记

September 29, 2025 · 2 min · 225 words

在 Linux 系统中，“一块 UFS”（通常指 Universal Flash Storage 这种闪存存储设备，在 Linux 下表现为一个块设备，如 /dev/sda 或 /dev/nvme0n1）被分区成多个分区后，监测系统的 IO 的含义是：

对存储子系统（block layer）中所有块设备进行读写操作的统计和观测，包括物理设备层面和逻辑分区层面。

1. 核心概念：什么是“IO”#

  • IO = Input/Output，这里特指块设备（block device）的 I/O 操作，即数据从内存到存储设备（或反之）的读写请求。
  • UFS 本身是物理硬件（闪存控制器 + NAND 颗粒），它被分区后：
    • 整个 UFS 设备对应 /dev/sda（或类似）。
    • 每个分区对应 /dev/sda1、/dev/sda2……（逻辑块设备）。
  • 所有分区共享同一套物理硬件资源（相同的控制器、相同的闪存通道、相同的队列），因此物理层面只有一个真实的 IO 通道。

2. “监测系统的 IO” 到底监测什么#

Linux 内核在 block layer 层面为每个块设备（无论物理盘还是分区）都维护独立的统计信息。你监测的“系统 IO”通常包含以下指标（常用工具 iostat -x、iotop、sar -d、fio、blktrace 等都可以看到）：

监测维度具体含义对应哪个设备？实际意义
tps / IOPS每秒完成的 IO 操作次数（read + write）可选整个 UFS 或单个分区反映随机读写能力
kB_read/s
kB_wrtn/s		每秒读取/写入的数据量（吞吐量）可选整个 UFS 或单个分区反映顺序读写带宽
%util设备繁忙时间百分比（队列非空时间占比）通常看整个 UFS最重要：>80% 就接近 IO 饱和
await / r_await / w_await每个 IO 请求的平均等待时间（ms）可选整个 UFS 或单个分区反映延迟
avgqu-sz平均 IO 队列长度可选整个 UFS 或单个分区队列越长，IO 压力越大
svctm设备自身服务每个 IO 的平均时间整个 UFS硬件本身处理能力

关键点：

  • 当你用 iostat -x 1 看 /dev/sda 时，显示的是整个 UFS 的总 IO（所有分区 IO 之和 + 内核元数据 IO）。
  • 当你指定看 /dev/sda1 时，显示的是该分区单独的 IO。
  • 但物理瓶颈永远在整个 UFS 设备上：即使某个分区 IO 很低，只要总 IO 把 %util 打满，整个系统（包括其他分区）都会变慢。

3. 为什么分区后“系统 IO”还是以物理设备为主？#

因为 Linux 的 IO 路径是： 应用程序 → 文件系统（ext4/xfs/btrfs 等） → 块层（bio） → UFS 驱动 → 硬件。

分区只是逻辑划分，所有 bio 请求最终都会汇集到同一个物理队列（mq-deadline / bfq / none 等）。所以“系统 IO”最核心的指标永远是整个 UFS 设备的 %util 和 await，而不是某个分区的数值。

4. 实际监测时的推荐做法#

  • 日常监控系统 IO：iostat -x 1 或 iotop -o 或 sar -d 1，重点看整个 UFS 设备（sda/nvme0n1）的 %util 和 await。
  • 定位哪个分区占 IO：iotop（按进程/分区看）或 blktrace + btt 分析。
  • 压力测试：用 fio 指定不同分区或整个设备，观察总 IO 是否达到 UFS 的规格（例如 UFS 3.1 典型 800MB/s 顺序读、几十万 IOPS）。

总结一句话：
“监测系统的 IO” 就是监测这块 UFS 物理设备（以及其上所有分区）在 Linux 块设备层面的真实读写压力。分区只是逻辑概念，物理 IO 永远是共享的，总 IO 才是判断系统是否有 IO 瓶颈的根本依据。
