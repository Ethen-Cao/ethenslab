## 1. 为什么车载屏需要 HDR

你可能有过这样的体验：在阳光强烈的白天，车载屏幕上的导航地图亮部发白，暗处一片模糊，视频画面失去层次感。这是因为传统显示技术（SDR†）的亮度表达范围有限，无法还原现实世界中从暗到亮的丰富层次。

HDR† 技术的引入，就是为了让屏幕能表达更宽的亮度范围——高光处不溢出发白，暗部保留细节，整体画面更接近人眼在真实场景中的观感。

---

## 2. HDR 是什么——用一句话理解

> **HDR（高动态范围）= 让画面的亮暗层次更丰富，让颜色范围更广**

它由两个维度共同构成：

| 维度 | 描述 | 对应标准 |
|------|------|---------|
| **动态范围**（亮暗） | 从最暗到最亮能表达多少层次 | PQ† 传递函数，上限 10000 nit |
| **色域†**（颜色范围） | 能呈现多少种颜色 | Rec.2020†（信令层），实际内容通常 P3† |

两者相互独立又相互配合。HDR 规范本身**不限制显示设备必须达到某个峰值亮度**，只规定了 metadata† 的格式；400 nit 的屏幕同样可以是"支持 HDR 的屏幕"，只是体验质量有差异。

## 3. 端到端链路：HDR 从内容到眼睛

HDR 不是一个开关，而是一条**端到端的链路**。就像一条水管，任何一节漏水，水都到不了目的地。

```
HDR 内容源
    │  携带 HDR metadata（MaxCLL、MaxFALL 等亮度信息）
    │  色深 ≥ 10-bit，使用 PQ 或 HLG 传递函数
    ▼
硬件解码器（VPU）
    │  解码 HEVC Main10 / AV1 10-bit 等 HDR Profile
    │  解析并传递 HDR 静态/动态 metadata
    ▼
GPU / 合成器
    │  FP16 或 10-bit 渲染管线
    │  处理 HDR 图层，识别正确色彩空间（BT.2020 + ST.2084）
    ▼
色调映射（TMO†）
    │  将内容的高动态范围映射到屏幕实际能显示的范围
    │  防止高光溢出或暗部丢失
    ▼
显示接口
    │  HDMI 2.0 / DisplayPort 1.4 / MIPI DSI
    │  携带 HDR InfoFrame / Metadata Packet 告知屏幕
    ▼
HDR 显示屏
    　  按正确的 EOTF† 曲线还原亮度
```

链路中任何一环缺失，HDR 都无法完整呈现。这也是为什么"芯片支持 HDR"和"系统支持 HDR"是两回事。

## 4. 高通平台 HDR 链路详解

### 4.1 硬件层

**视频解码器（VPU）**

高通座舱芯片内置专用视频处理单元，负责将压缩的视频码流还原为图像数据。要支持 HDR，VPU 需要：

- 支持 HEVC（H.265）Main10 Profile —— 这是当前 HDR 视频最主流的编码格式
- 支持 VP9 Profile 2 和 AV1 10-bit —— 用于流媒体和 OTT 内容
- 能读取并向上层传递 HDR 静态 Metadata（SEI 中的 mastering display color info 和 content light level info）

**GPU（Adreno）**

Adreno GPU 负责界面渲染、OpenGL/Vulkan 应用绘制、图层合成等工作。HDR 渲染要求：

- 支持 RGBA1010102 格式的 framebuffer†（每通道 10-bit）
- 支持 FP16（16-bit 浮点）帧缓冲，用于高精度色彩计算
- Adreno 660（SA8155P）和 Adreno 735（SA8295P）均已具备上述能力

**显示处理单元（DPU / MDP）**

DPU 是芯片内负责最终图像输出的模块，相当于"图像的最后一公里"。其 HDR 能力包括：

- 支持 10-bit 输出管线
- 内置硬件色调映射模块（TMO），能处理 PQ 和 HLG 两种传递函数
- 能将 HDR Metadata 转发至下游显示接口

**显示接口**

- **HDMI 2.0b**：通过 InfoFrame 携带 HDR 信息（包含 EOTF 类型、色域参数等），屏幕据此知道"这是 HDR 内容"
- **DisplayPort 1.4**：通过 HDR Metadata Packet 传递
- **MIPI DSI**：连接车载内嵌屏，通过 10-bit 色深 + DSC 压缩传输

---

### 4.2 驱动与 HAL 层

驱动层是硬件能力暴露给软件的桥梁。HDR 支持需要以下驱动配合：

**内核 DRM/KMS 驱动**

Linux 内核的显示子系统（DRM/KMS）通过属性机制管理显示能力。HDR 支持需要驱动实现 `HDR_OUTPUT_METADATA` property，让合成器可以将 HDR 元数据下发给硬件。高通的 MDSS（Mobile Display Subsystem）驱动需要显式开启 HDR pipeline。

**Display HAL（HWC2/HWC3）**

HAL（硬件抽象层）是 Android 系统框架与驱动之间的接口。HDR 相关的关键接口：

- `getHdrCapabilities()`：上报屏幕支持哪些 HDR 类型（HDR10、HLG、Dolby Vision 等）
- `setLayerDataspace()`：通知硬件合成器当前图层的色彩空间
- HDR layer 的合成调度：决定哪些图层走硬件 HDR 路径，哪些需要软件 TMO 降级

**QDCM（Qualcomm Display Color Management）**

高通专有的色彩管理模块，负责配置屏幕的色彩校准 Profile，包括屏幕白点、色域、亮度特性等。这些参数直接影响 DPU 的 TMO 计算是否准确。

---

### 4.3 软件框架层

**SurfaceFlinger（Android 合成器）**

SurfaceFlinger 是 Android 系统的图层合成服务，相当于"图像的导演"。HDR 支持需要：

- 识别图层的 Dataspace 标记（例如 `BT2020_PQ` 表示这是 HDR10 内容）
- 当 HDR 图层与普通 SDR 图层同屏显示时，正确处理两者的亮度混合（HDR to SDR tone mapping）
- 系统属性 `ro.surface_flinger.has_HDR_display=true` 需要设置为 true

> 非 Android 平台（如基于 Weston 的 Linux 系统）则需要相应的 Wayland 合成器实现类似能力。

**媒体框架**

Android MediaCodec 解码 HDR 视频后，输出格式为 `COLOR_FormatYUVP010`（10-bit YUV），同时通过 `MediaFormat` 的 `KEY_HDR_STATIC_INFO` 字段携带 HDR Metadata，传递给 SurfaceFlinger 用于后续处理。

**色彩空间传递链**

整个软件栈需要对色彩空间信息做到"一路保真"：解码器输出标记 BT.2020 + PQ → SurfaceFlinger 识别并保留标记 → DPU 按照标记选择正确的处理路径 → 显示接口输出正确的 InfoFrame。任何一环丢失色彩空间信息，都会导致画面颜色异常。

---

### 4.4 应用层

对于视频播放类应用：
- 使用 MediaCodec 解码，设置支持 HDR10 的 Surface，并正确传递 `HDR_STATIC_INFO`
- 播放器需处理 HDR 内容的 MaxCLL/MaxFALL 信息，决定是否需要应用层 TMO

对于 UI 渲染类应用：
- 使用 RGBA1010102 或 FP16 格式的 Surface 进行 HDR UI 渲染
- 正确设置 `ColorSpace` 为 `BT2020 PQ` 或 `DISPLAY_P3`

## 5. 整体依赖总结

```
┌─────────────────────────────────────────────────────┐
│                    应用 / 播放器                      │
│         传递 HDR Metadata + 10-bit Buffer             │
├─────────────────────────────────────────────────────┤
│              媒体框架 / SurfaceFlinger                │
│       识别 HDR Dataspace，管理 HDR/SDR 层混合          │
├─────────────────────────────────────────────────────┤
│         HWC HAL / Display HAL（高通 SDM）             │
│         上报 HDR 能力，调度 HDR pipeline              │
├──────────────┬──────────────────┬───────────────────┤
│  VPU 硬解    │   Adreno GPU     │   DPU 显示引擎     │
│ HEVC Main10  │ FP16/RGBA1010102 │ 10-bit + TMO 模块  │
├──────────────┴──────────────────┴───────────────────┤
│          显示接口（HDMI 2.0 / DP 1.4 / DSI）          │
│              HDR InfoFrame / Metadata Packet          │
├─────────────────────────────────────────────────────┤
│                  HDR 车载显示屏                        │
│           按 PQ/HLG 正确还原亮度，声明 HDR 支持        │
└─────────────────────────────────────────────────────┘
```

**关键原则**：任何一层缺失，HDR 都无法完整工作。"芯片支持 HDR"只是必要条件，完整的 HDR 体验需要上图中每一层都正确配置。

## 6. 常见问题与坑

**Q：芯片选型支持 HDR，但实际画面颜色偏黄/偏绿？**  
通常是色彩空间传递链断裂。检查 SurfaceFlinger 的 Dataspace 设置是否正确传递到 DPU，以及 QDCM 的屏幕 Profile 是否与实际面板特性匹配。

**Q：HDR 视频播放正常，但 UI 叠加后高光溢出？**  
HDR 图层与 SDR UI 图层混合时，需要合成器对 SDR 层做亮度提升（SDR boost）或对 HDR 层做 tone mapping，使两者亮度匹配。检查 SurfaceFlinger 的 HDR/SDR 混合逻辑。

**Q：换了一块新屏幕，HDR 失效？**  
新屏幕需要在 EDID（HDMI 接口）或 panel timing 配置文件中声明 HDR 能力；同时 QDCM 的屏幕 Profile 需要对应更新。如果屏幕本身不支持 HDR，系统应自动触发 HDR→SDR tone mapping 降级输出。

**Q：同一块屏，有时 HDR 生效有时不生效？**  
常见原因：HDR Metadata 在某些播放路径下没有被传递（例如通过 SurfaceView 但没有正确设置 ColorSpace），导致系统误判为 SDR 内容。

## 附录：名词解释

---

### SDR（Standard Dynamic Range，标准动态范围）

**是什么**：SDR 是 HDR 出现之前的传统图像标准，定义了一套亮度范围和颜色空间的规范。

**历史**：SDR 标准（Rec.709）诞生于 1990 年代，为高清电视设计。当时假设显示设备的峰值亮度约为 100 nit（1 nit ≈ 1 cd/m²，烛光/平方米），这在当时是合理的，但随着显示技术发展，现代屏幕亮度早已远超这一数值。

**局限**：SDR 的亮度范围约为 0.1～100 nit，这意味着明亮的窗户、阳光等场景在画面中会直接"溢出"成一片白，暗部细节也因精度不足而丢失。

---

### HDR（High Dynamic Range，高动态范围）

**是什么**：HDR 是一系列技术标准的统称，核心目标是让数字图像能表达更宽的亮度范围和更广的色域，使画面更接近人眼的真实感知。

**历史**：HDR 的概念最早来自摄影领域（通过多次曝光合成），2013 年前后逐步进入消费电子领域。2015 年 Ultra HD Blu-ray 标准发布，HDR10 成为主流规范，随后各厂商推出了多个衍生标准。

**主要标准对比**：

| 标准 | 传递函数 | 色深 | Metadata 类型 | 备注 |
|------|---------|------|--------------|------|
| HDR10 | PQ（ST.2084） | 10-bit | 静态（全片一组） | 最广泛支持 |
| HDR10+ | PQ | 10-bit | 动态（逐帧/逐场景） | 三星主推，开放协议 |
| Dolby Vision | PQ | 12-bit | 动态 | 杜比私有，授权费用 |
| HLG | HLG 曲线 | 10-bit | 无 | 广播级，兼容 SDR 显示 |

**重要澄清**：HDR 标准本身只规定 metadata 的格式（携带哪些亮度信息），**不限制显示设备必须达到的峰值亮度**。400 nit 的屏幕能正确解析 HDR 信号就是"支持 HDR 的屏幕"，只是相比 1000 nit 的屏幕体验质量更低。

---

### PQ（Perceptual Quantizer，感知量化器）

**是什么**：PQ 是 HDR 内容使用的亮度编码曲线（传递函数），正式名称为 SMPTE ST.2084。

**原理**：人眼对亮度的感知是非线性的——在暗处，非常微小的亮度变化就能被察觉；在亮处，需要很大的亮度变化才能被感知到。PQ 曲线模拟了这一特性，将更多的编码精度分配给人眼敏感的亮度区间，从而在 10-bit 的数据量内表达极宽的亮度范围。

**上限**：PQ 曲线的设计上限是 **10000 nit**。这不是说显示屏必须达到 10000 nit，而是定义了编码能力的天花板，确保未来更亮的显示设备也在同一标准框架内工作。

**对比 SDR 的 gamma 曲线**：传统 SDR 使用 gamma 2.2 曲线（约 100 nit 上限），PQ 把上限提升了约 100 倍。

---

### HLG（Hybrid Log-Gamma，混合对数伽马）

**是什么**：HLG 是另一种 HDR 传递函数，由 BBC 和 NHK 联合开发，专为广播电视设计。

**特点**：HLG 的聪明之处在于——同一条 HLG 信号，在 SDR 显示器上看是正常的 SDR 画面，在 HDR 显示器上看自动呈现 HDR 效果，无需任何额外处理。这种向下兼容能力使其非常适合直播场景。

**与 PQ 的主要区别**：PQ 携带绝对亮度信息（告诉显示器"这里是多少 nit"），HLG 使用相对亮度信息（让显示器自行决定如何映射）。

---

### 色域（Color Gamut）

**是什么**：色域描述一个设备（摄像机、屏幕、打印机）能表达的颜色范围。在 CIE 色度图上，色域通常表示为一个三角形，三个顶点是红、绿、蓝三原色的位置，三角形越大代表能表达的颜色越多。

**常见色域对比**：

| 色域 | 覆盖范围（相对人眼可见色） | 主要应用 |
|------|----------------------|---------|
| sRGB / Rec.709 | 约 35% | 电脑显示器、网络图片、SDR 视频 |
| DCI-P3 | 约 45% | 数字电影院、苹果设备 |
| Rec.2020 | 约 75% | HDR 视频标准，未来目标 |

**HDR 与色域的关系**：HDR10 在信令层使用 Rec.2020 作为色彩空间容器，但现实情况是当前大多数制作的 HDR 内容母版是在 DCI-P3 色域完成的。Rec.2020 更多是一个"信封"，保证了标准的前向兼容性。

---

### 色彩空间（Color Space）

**是什么**：色彩空间是一套完整描述颜色的数学框架，包含色域（能表达哪些颜色）、白点（白色在哪里）、传递函数（亮度如何编码）三个要素。

**例子**：
- `sRGB` = Rec.709 色域 + D65 白点 + gamma 2.2 传递函数
- `BT.2020 PQ` = Rec.2020 色域 + D65 白点 + PQ 传递函数（= HDR10 的色彩空间）
- `Display P3` = P3 色域 + D65 白点 + sRGB 传递函数（= 苹果设备常用）

在软件栈中，色彩空间通过"Dataspace"标记随图像数据一起传递，让各层模块知道如何正确处理这块数据。

---

### Metadata（元数据）

**是什么**：Metadata 直译为"描述数据的数据"。在 HDR 语境中，metadata 是附在视频内容旁边的一组参数，描述这段内容的亮度特性，帮助显示设备做出最佳的色调映射决策。

**主要字段**：

| 字段 | 含义 |
|------|------|
| MaxCLL | 整段内容中单个像素的最大亮度（Content Light Level） |
| MaxFALL | 整段内容中帧平均最大亮度（Frame Average Light Level） |
| Mastering Display Primaries | 制作时使用的监视器色域 |
| Mastering Display Min/Max Luminance | 制作时监视器的最暗/最亮值 |

**静态 vs 动态 Metadata**：
- **静态 Metadata**（HDR10）：整段视频用同一组参数，简单但精度不足——一部电影里最亮的爆炸场景和最暗的夜景用同一套映射参数，效果必然有妥协
- **动态 Metadata**（HDR10+、Dolby Vision）：逐场景甚至逐帧变化，针对性更强，效果更好

---

### TMO（Tone Mapping Operator，色调映射）

**是什么**：TMO 是将一个动态范围的图像映射到另一个动态范围的算法。最常见的场景是将 HDR 内容（可能达到 1000～10000 nit）映射到实际屏幕能显示的范围（例如 400 nit）。

**为什么需要 TMO**：就像把一张 A3 的画缩印到 A4 纸上——不能直接截掉，要合理缩放，同时保留重要细节。好的 TMO 能在压缩亮度范围的同时，保留画面的对比度感知和色彩关系。

**在高通平台中的位置**：TMO 可以发生在多个层级——DPU 硬件模块、SurfaceFlinger 软件合成、或应用层。当 HDR 图层与 SDR 图层混合时，TMO 尤其关键。

---

### EOTF（Electro-Optical Transfer Function，电光传递函数）

**是什么**：EOTF 描述"数字信号值"到"屏幕发出的光亮度"的映射关系。简单说，就是屏幕拿到一个数字 100，它应该发出多亮的光？

**为什么重要**：内容制作时按照某个 EOTF 编码，显示时必须按照同样的 EOTF 解码，否则画面会偏亮或偏暗。HDR InfoFrame 中携带的 EOTF 信息，就是告诉屏幕"请用哪条曲线来还原亮度"。

---

### DCI-P3（P3 色域）

**是什么**：DCI-P3 是数字电影工业协会（Digital Cinema Initiatives）定义的色域标准，比 sRGB 宽约 25%，特别是绿色和红色更为饱和鲜艳。

**历史与应用**：最初为数字电影院设计（约 2005 年），后被苹果公司引入消费设备（2015 年 iMac Retina 5K 是首款支持 P3 的苹果电脑），现已成为高端手机、平板、笔记本屏幕的主流色域标准。

**与 HDR 内容的关系**：虽然 HDR10 的信令使用 Rec.2020，但当前绝大多数 HDR 内容的实际制作是在 P3 色域监视器上完成的，内容真正覆盖的色彩范围是 P3 而非完整的 Rec.2020。

---

### Framebuffer（帧缓冲）

**是什么**：Framebuffer 是 GPU 在内存中维护的一块区域，存储最终准备输出到屏幕的像素数据。可以理解为"即将印刷的版面"。

**与 HDR 的关系**：传统 SDR 系统使用 8-bit per channel 的 RGBA8888 格式（每个颜色通道 0～255 共 256 级），精度对 HDR 内容不够。HDR 需要 RGBA1010102（每通道 10-bit，1024 级）或 FP16（16-bit 浮点）格式的 framebuffer，以保留 HDR 的亮度精度。

---

### HAL（Hardware Abstraction Layer，硬件抽象层）

**是什么**：HAL 是 Android 系统中连接上层软件框架（如 SurfaceFlinger）与底层硬件驱动的标准接口层。它定义了一组标准 API，让上层不需要关心具体芯片的实现细节。

**Display HAL 与 HDR 的关系**：Display HAL（HWC，Hardware Composer）负责告诉上层"这块屏幕支持哪些 HDR 类型、最大亮度是多少"，同时接收上层的 HDR 图层合成指令，转化为具体的硬件操作。
