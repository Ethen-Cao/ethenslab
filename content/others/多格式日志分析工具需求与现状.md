# 多格式日志分析工具需求与现状

## 1. 背景

当前 `/home/ethen/bin/logcat` 适合处理 Android `logcat threadtime` 风格日志，但已经无法覆盖测试现场实际给到的日志集合。测试提供的输入可能是单个文件、文件夹、压缩包，内部可能混合 Android log、Linux 用户态日志、DLT 文本、二进制 DLT、`dmesg`、boot/UEFI 串口日志等多种格式。

因此需要先明确需求边界、当前样本现状和后续设计约束，再进入工具架构讨论。

## 2. 明确需求

### 2.1 输入形态

工具需要支持以下输入入口：

- 单个文件
- 单个目录
- 压缩包
- 压缩包内继续嵌套的日志文件

输入处理层不能假设“日志已经被人工解压并整理好”。

### 2.2 日志格式

工具需要支持或可扩展支持以下日志族：

- Android log
- Linux log
- `dmesg` log
- DLT 文本日志
- 二进制 DLT 日志
- boot/UEFI/串口类原始日志

这里不能只抽象成 “Android” 和 “Linux” 两类，因为 Linux 侧本身就存在多种时间格式、字段语义和封装层次。

### 2.3 查询能力

测试在提 issue 时，通常会给出“某个时间点发生了什么问题”的描述，因此工具第一目标不是关键词搜索，而是时间范围解析正确。

第一阶段至少需要支持：

- 时间范围过滤
- `source` 过滤
- `keyword` 包含过滤
- `exclude` 排除过滤
- `pid` 过滤

后续大概率还会扩展：

- `level`
- `tag`
- `app`
- `ctx`
- `tid`
- `boot session`

### 2.4 Boot Session

工具需要检测系统开机次数，并识别每次运行的时间段。当前阶段先以 Android 日志为主来切分 boot session，但模型本身需要支持未来接入其他来源。

除 boot session 切分外，还需要把每次 boot 对应的 `dmesg` 日志关联起来。

## 3. 当前样本现状

### 3.1 当前样本目录

本次分析样本位于：

`/home/ethen/Downloads/#226实车同版本刷写失败U盘日志2026_03_26_18_51_15/2026_03_26_18_51_15/linux_log`

### 3.2 当前样本文件形态

当前目录并不是单一日志格式，而是混合输入：

| 类型 | 数量 | 说明 |
|------|------|------|
| `.txt` | 58 | 包含 DLT 文本、journal 文本等 |
| `.dlt` | 57 | 二进制 DLT |
| `.gz` | 3 | 压缩 DLT |
| `.tgz` | 6 | 打包 DLT 分片 |
| `.log` | 7 | Unity/业务日志 |
| 无标准扩展名轮转文件 | 6 | `la_gvm.txt._*` 一类轮转产物 |

说明：

- 当前样本中没有直接命名为 `dmesg` 或 `kmsg` 的文件。
- `dltlog/dlt/*.tgz` 内部仍是 `.dlt` 文件，说明输入层必须支持“容器内嵌日志”。

### 3.3 当前样本里的主要日志族

| 日志族 | 样例 | 时间形态 | 可提取字段 | 现状判断 |
|------|------|------|------|------|
| DLT 文本 | `dltlog/dlt_118.txt` | `YYYY/MM/DD HH:MM:SS.ffffff` | `app/ctx/level/message`，部分消息内含 `file,line` | 可作为第一批支持对象 |
| DLT journal 文本 | `dltlog/journal_178.txt` | 外层 `YYYY/MM/DD HH:MM:SS.ffffff`，内层消息也带时间 | `ecu/app/ctx/level`，内层可抽 `proc[pid]` | 需要双层时间与双层 source 抽象 |
| Unity 日志 | `tuanjie/LogFiles/PlayerLog7.log` | `YYYY-MM-DD HH:MM:SS.mmm` | `level/source/message`，线程信息有限 | 可作为第一批支持对象 |
| 二进制 DLT | `dltlog/*.dlt` | 需解码后才能识别 | 取决于解码器 | 不应直接交给文本 parser |
| 压缩 DLT | `dltlog/*.gz`、`dltlog/dlt/*.tgz` | 需先解压/解包 | 取决于解码器 | 输入层必须单独处理 |
| boot/UEFI 原始日志 | `la_gvm/la_gvm.txt` | 大量行无标准 wall-clock 时间 | 仅部分行有计数或阶段文本 | 更接近 raw/boot log，不应套用 Android/DLT 规则 |

### 3.4 现有 `logcat` 脚本的局限

现有脚本本质上是 Android `threadtime` 格式过滤器，而不是通用日志查询器。

关键局限：

- 把 `$1` 固定假设为 `MM-DD`
- 把 `$2` 固定假设为 `HH:MM:SS.xxxxxx`
- 把固定列位当成 `pid/tag`
- 时间归一化逻辑仅覆盖 Android 风格

因此在当前样本目录里直接执行 `bash /home/ethen/bin/logcat -k error .` 会得到空结果，因为大多数 Linux 侧文本日志在格式校验阶段就被丢弃了。

## 4. 设计输入与约束

### 4.1 时间必须被建模成一等公民

时间不是普通字段，而是主查询轴。设计上至少要区分：

- 绝对 wall-clock 时间
- 相对开机时间
- 无法识别或缺失的时间
- 同一条日志里的外层时间和内层时间

工具内部应同时保留：

- 原始时间字符串
- 归一化后的比较键
- 时间类型
- 时间来源

否则后续做跨格式排序、boot session 切分、`dmesg` 关联时会出现语义混乱。

### 4.2 查询语义不能再绑定某一种格式

现有脚本是围绕 Android `pid/tag/keyword` 写死的。后续需要扩展到 DLT、journal、Unity、`dmesg` 时，查询模型必须脱离固定列位。

建议后续统一成以下思路：

- 同一字段多个值按 OR 匹配
- 不同字段之间按 AND 匹配
- `exclude` 永远作为全局排除条件
- 时间范围始终先于内容匹配

### 4.3 输入层和解析层必须拆开

后续架构至少要分成两层：

- 输入层：负责打开文件、递归目录、解压压缩包、解出容器内文件
- 解析层：负责识别日志格式，并把每一行转换成统一事件模型

否则二进制 DLT、`.tgz`、`.gz` 一进来就会把 parser 逻辑污染掉。

## 5. Boot Session 与 `dmesg` 关联问题

### 5.1 当前需求

需要回答两个问题：

- 系统一共 boot 了几次
- 每次 boot 对应哪个 `dmesg` 文件或 `dmesg` 片段

### 5.2 当前阶段建议

boot session 切分当前先以 Android 日志为准，这是合理的，因为 Android 日志通常更容易拿到绝对时间，也更容易识别“从开机到系统可用”的阶段边界。

但 `dmesg` 关联如果只靠模糊时间匹配，长期一定会出错。真正可靠的方案不是“去猜一条相似事件”，而是显式建立 boot correlation key。

### 5.3 最强建议：引入 `boot_id`

如果允许改采集链路，我最建议把 Linux 的 `boot_id` 当成 boot session 的主关联键，而不是依赖某条业务事件。

建议方式：

- 在系统启动后尽早读取 `/proc/sys/kernel/random/boot_id`
- 在 Android 侧尽早打一次带 `boot_id` 的日志
- 在 `dmesg` 导出文件名、头部或旁路 metadata 中记录同一个 `boot_id`

这样 Android boot session 和 `dmesg` 的关联可以从“概率匹配”变成“精确匹配”。

### 5.4 如果当前样本里没有 `boot_id`

如果历史包里没有 `boot_id`，只能做降级关联。建议按如下优先级组合使用：

- boot 开始时间窗口
- reboot reason / boot reason
- `init` 第二阶段启动类事件
- Android `boot_completed` 或系统 ready 事件
- `dmesg` 文件采集时间、文件名时间、打包时间
- 内核版本、slot、build fingerprint、设备序列等上下文信息

注意：

- `boot_completed` 适合判断 boot session 的完成，但不适合作为 `dmesg` 唯一关联锚点。
- 单独依赖“某条业务日志”并不稳，因为业务进程可能起不来、可能被裁剪、也可能跨多个 boot 重复出现。

### 5.5 如果必须在“事件”里选一个锚点

如果不允许引入 `boot_id`，又必须在现有日志事件里找一个最强锚点，我建议优先考虑：

- Android 侧早期 `init` 启动阶段事件
- 同一 boot 的 `reboot reason` / `boot reason`

原因：

- 这些事件离 boot 起点更近
- 更接近系统级而不是业务级
- 在 Android 日志和内核/引导链路中更可能留下对应痕迹

但这里仍然建议把“单事件锚点”理解为降级方案，而不是最终架构。

## 6. 后续设计讨论应回答的问题

在进入具体实现前，后续设计文档需要继续回答以下问题：

- 统一事件模型需要有哪些字段
- 时间模型如何表达绝对时间、相对时间、缺失时间
- boot session 的 start/end 判据如何配置化
- `dmesg` 关联规则如何从“精确键”降级到“启发式”
- 输入层是否要支持流式解压和增量解析
- 二进制 DLT 采用什么解码方案
- 工具主实现语言是否改为 Python，并把 shell 只保留为薄入口

## 7. 当前结论

当前问题已经不是“给 `logcat.bash` 多加几个正则”可以解决的，而是要把工具升级成面向多输入、多格式、多 boot session 的日志分析框架。

第一优先级应是：

1. 建立正确的时间模型
2. 建立 boot session 模型
3. 建立 Android boot session 与 `dmesg` 的稳定关联键
4. 在此基础上再扩展 `source/keyword/exclude/pid` 等查询能力

## 8. 设计文档

### 8.1 设计目标

工具的设计目标按优先级排序如下：

1. 时间解析正确，且错误时可解释
2. boot session 切分正确，且证据可追溯
3. 支持多输入形态和多日志格式
4. 查询语义跨格式保持一致
5. 新格式接入成本低，不需要改核心查询逻辑

核心原则：

- 不再围绕某一种日志列位设计
- 不再把“解包、识别、解析、过滤、展示”混写在一个脚本里
- 对所有推断型结果保留 evidence，而不是只输出结论

### 8.2 技术选型建议

建议使用 Python 作为主实现语言，shell 只保留为薄入口或兼容入口。

原因：

- 需要处理目录、压缩包、二进制、文本、递归输入，Python 更适合建模
- 需要实现格式检测、统一事件模型、boot session 状态机，shell 可维护性太差
- 后续如果要接入 DLT 解码器、SQLite、JSON 导出、测试框架，Python 扩展成本更低

建议实现形态：

- 主程序：Python CLI
- 兼容层：保留 `logcat` 命令名，但内部转调 Python
- shell：仅负责参数透传，不再承载核心逻辑

这里需要额外明确一条约束：

- 第一阶段的新工具必须先覆盖现有 `logcat` 脚本已具备的用户能力，不能为了做架构升级而让已有工作流退化

### 8.3 总体架构

建议采用如下分层：

```text
CLI / API
    ->
Query Parser
    ->
Input Resolver
    ->
Artifact Inventory
    ->
Format Detector
    ->
Parser Plugins
    ->
Normalized Event Stream
    ->
Boot Session Detector
    ->
Association Engine
    ->
Query Engine
    ->
Renderer / Exporter
```

各层职责如下：

| 层 | 职责 | 是否感知具体日志格式 |
|------|------|------|
| CLI / API | 接收参数、输出结果 | 否 |
| Input Resolver | 处理文件、目录、压缩包、递归容器 | 否 |
| Artifact Inventory | 为每个逻辑输入分配稳定 ID 和元信息 | 否 |
| Format Detector | 对每个文件判断格式 | 是 |
| Parser Plugins | 把原始记录转成统一事件 | 是 |
| Boot Session Detector | 从事件流中识别 boot 边界 | 否 |
| Association Engine | 关联 Android boot 与 `dmesg`/其他工件 | 否 |
| Query Engine | 按统一字段做过滤、排序、聚合 | 否 |
| Renderer | 文本/JSON/表格输出 | 否 |

### 8.4 核心对象模型

建议至少定义以下核心对象。

#### 8.4.1 Artifact

代表一个“可被解析的逻辑输入对象”，例如：

- 一个普通文本文件
- 一个 `.gz` 解出的文件
- 一个 `.tgz` 里的某个成员
- 一个二进制 `.dlt` 文件

建议字段：

| 字段 | 含义 |
|------|------|
| `artifact_id` | 稳定唯一 ID |
| `container_id` | 外层容器 ID，顶层为空 |
| `origin_path` | 用户给定的原始路径 |
| `logical_path` | 解包后的逻辑路径，如 `foo.tgz::bar/dlt_001.dlt` |
| `file_type` | 文本、二进制、压缩包成员等 |
| `size` | 大小 |
| `mtime` | 文件修改时间 |
| `sniff_bytes` | 格式检测使用的样本字节 |
| `metadata` | 采集时间、包名、目录名时间戳等上下文 |

#### 8.4.2 Event

代表被 parser 归一化后的一条日志事件。

建议字段：

| 字段 | 含义 |
|------|------|
| `event_id` | 全局唯一 ID |
| `artifact_id` | 来源工件 |
| `parser` | 解析器名称 |
| `line_no` / `offset` | 原始位置 |
| `raw` | 原始文本或摘要 |
| `message` | 解析后的主消息 |
| `level` | 标准化级别 |
| `pid` / `tid` | 进程/线程 |
| `tag` | Android tag |
| `app` / `ctx` | DLT APID/CTID |
| `proc` | 进程名或 source 进程 |
| `source` | 跨格式统一 source |
| `wall_time` | 外层绝对时间 |
| `inner_time` | 内层时间，如 journal 内嵌时间 |
| `mono_time` | 相对开机时间，如 `dmesg` 秒数 |
| `boot_id` | 若可得则记录 |
| `boot_session_id` | 归属 boot session |
| `evidence` | 解析和推断证据 |

#### 8.4.3 BootSession

代表一次系统运行周期。

建议字段：

| 字段 | 含义 |
|------|------|
| `boot_session_id` | 唯一 ID |
| `start_time` | 绝对开始时间，可能为空 |
| `end_time` | 绝对结束时间，可能为空 |
| `start_mono` | 相对时间起点，通常为 0 |
| `boot_id` | 若可得则记录 |
| `boot_reason` | 若可得则记录 |
| `source_artifacts` | 支撑该 session 的工件集合 |
| `confidence` | `confirmed/inferred/partial` |
| `start_evidence` | 起点证据 |
| `end_evidence` | 终点证据 |
| `linked_dmesg` | 关联到的 `dmesg` 工件或片段 |

### 8.5 时间模型

时间模型是本工具的第一优先级，建议单独建模，而不是只存一个字符串。

#### 8.5.1 时间类型

每个时间字段都要区分类型：

- `wall_clock`: 绝对时间，例如 `2026/03/26 18:50:26.239051`
- `wall_clock_partial`: 绝对时间不完整，例如 Android `MM-DD HH:MM:SS.mmm`
- `monotonic`: 相对开机时间，例如 `dmesg` 的 `[ 12.345678 ]`
- `sequence_only`: 只有顺序没有时间
- `missing`: 没有可用时间

#### 8.5.2 时间对象建议字段

| 字段 | 含义 |
|------|------|
| `raw` | 原始时间字符串 |
| `kind` | 时间类型 |
| `epoch_ns` | 绝对时间的归一化值 |
| `mono_ns` | 相对时间的归一化值 |
| `timezone` | 时区 |
| `precision` | 秒、毫秒、微秒、纳秒 |
| `confidence` | `exact/inferred/unknown` |
| `source` | 来自外层前缀、内层正文、文件名、metadata 等 |
| `inferred_parts` | 被推断出来的部分，如年份 |

#### 8.5.3 时间归一化规则

建议规则如下：

1. 原始字符串永远保留
2. 可确定的绝对时间统一归一到 `epoch_ns`
3. `dmesg` 这类相对时间单独保留在 `mono_ns`
4. 不允许把相对时间直接伪装成绝对时间
5. 对缺少年份的 Android 时间，必须显式记录“年份推断来源”

#### 8.5.4 Android 年份推断

Android `threadtime` 常见问题是只有 `MM-DD HH:MM:SS.mmm`，没有年份。

建议推断优先级：

1. 同包内其他带完整日期的日志
2. 压缩包名或目录名里的采集日期
3. 文件名里的完整日期
4. 文件 `mtime`

如果以上来源仍然冲突或不可靠：

- 不要静默推断
- 将时间标记为 `wall_clock_partial`
- 对跨年或跨包比较发出 warning

这样才能满足“时间范围解析正确”的目标，而不是给出一个看似完整但无法证明正确的时间。

### 8.6 格式检测设计

格式检测单位必须是“文件”，不能按目录一次性判定。

建议检测优先级：

1. 用户显式指定 `--format`
2. 文件扩展名和 magic bytes
3. 前若干行内容特征
4. 父容器上下文
5. fallback 到 `raw_text` 或 `raw_binary`

建议首批检测器：

| 检测器 | 典型特征 |
|------|------|
| `android_threadtime` | 行首 `MM-DD HH:MM:SS` |
| `dlt_text` | 行首序号 + `YYYY/MM/DD HH:MM:SS.ffffff` + `ECU/APID/CTID` |
| `journal_text` | DLT 文本外层包裹内层 `proc[pid]` 消息 |
| `unity_playerlog` | `[Info]... YYYY-MM-DD HH:MM:SS.mmm :` |
| `dmesg_text` | 行首 `[ 12.345678 ]` 或 kernel 风格前缀 |
| `boot_raw` | 大量无统一时间，包含 bootloader/UEFI 关键词 |
| `dlt_binary` | 二进制 DLT magic 或扩展名 |

### 8.7 Parser 插件设计

每个 parser 只做一件事：把某种格式转换成统一 `Event`。

接口建议：

```python
class ParserPlugin:
    name: str

    def detect(self, artifact: Artifact) -> DetectionResult: ...

    def parse(self, artifact: Artifact, ctx: ParseContext) -> Iterator[Event]: ...
```

约束：

- parser 不负责解压容器
- parser 不负责跨文件排序
- parser 不负责最终查询
- parser 只负责单文件、单格式、单向转换

建议首批 parser：

- `AndroidThreadtimeParser`
- `DltTextParser`
- `DltJournalParser`
- `UnityPlayerLogParser`
- `DmesgTextParser`
- `BootRawParser`

### 8.8 `source` 字段统一策略

用户需要一个跨格式都能理解的 `source`，但不同格式里的 source 含义并不完全相同。

建议定义：

- Android：`source = tag`
- DLT 文本：`source = app`，同时保留 `ctx`
- journal 文本：`source = 内层 proc`，没有内层 proc 时退化为 `app/ctx`
- Unity：`source = logger name`，没有则退化为固定文件源
- `dmesg`：`source = kernel subsystem` 或解析出的前缀
- boot raw：`source = stage/component`

同时保留结构化字段：

- `tag`
- `app`
- `ctx`
- `proc`
- `pid`

这样：

- `source` 可作为通用入口
- 高精度分析仍可使用结构化字段

### 8.9 第一阶段与现有 `logcat` 的兼容目标

第一阶段不是“做出一个新框架雏形”就结束，而是要达到“至少能替代现有 `logcat` 脚本”的交付标准。

#### 8.9.1 必须兼容的 CLI 能力

第一阶段至少兼容现有 `logcat` 的这些能力：

- 支持多个搜索路径
- 支持默认搜索当前目录
- 支持 `-f`
- 支持 `-p`
- 支持 `-s`
- 支持 `-k`
- 支持 `-e`
- 支持 `-t`
- 支持 `--plain`
- 支持多 PID、多 source、多关键字、多排除词
- 支持带文件名和行号的输出

建议第一阶段保留 `logcat` 命令名和主要参数习惯，避免测试和分析同学切换成本过高。

#### 8.9.2 必须兼容的查询语义

现有 `logcat` 的核心语义是：

```text
Time AND (NOT Exclude) AND ( PID OR Tag OR Keyword )
```

第一阶段建议做两层兼容：

- 对外保持现有 `logcat` 的主要参数和使用方式
- 对内统一收敛到新的查询模型

为了兼容现有使用习惯，第一阶段建议提供一层 compatibility mapping：

- `-s` 在 Android 上等价于 `tag`
- `-s` 在其他格式上优先映射到统一 `source`
- `-p` 映射到 `pid`
- `-k` 映射到 `keyword include`
- `-e` 映射到 `exclude`
- `-t` 映射到时间范围

注意：

- 第一阶段内部引擎可以采用“同字段 OR、跨字段 AND”的统一模型
- 但对 `logcat` 兼容入口，需要保证用户感知结果不明显偏离旧脚本

#### 8.9.3 对不完全支持字段的处理原则

不同格式并不一定都能稳定提供 `pid`、`tag` 或 `source`。第一阶段要避免两种错误：

- 静默忽略用户条件
- 伪造一个并不真实的字段

建议策略：

- 字段可解析时正常过滤
- 字段天然不存在时输出明确 warning
- warning 不应中断查询
- 结果中需要标记哪些 artifact 没有参与某个字段过滤

#### 8.9.4 第一阶段的兼容边界

“实现同样的功能”不等于“逐字节复刻旧脚本实现”，而是要保证下列事实成立：

- 旧脚本能完成的常见查询，新工具第一阶段也能完成
- Android `threadtime` 查询体验不退化
- 现有参数和命令名尽量不变
- 时间过滤、keyword/include/exclude、PID/source 过滤能力不退化
- 输出可读性不退化

不要求第一阶段完全复制旧脚本的内部实现细节，例如：

- 不要求继续使用 `rg + awk`
- 不要求继续依赖 Android 固定列位
- 不要求继续保留旧脚本内部的偶然行为

### 8.10 查询引擎设计

查询语义建议统一为：

- 时间范围先过滤
- `exclude` 永远是全局否定条件
- 同字段多值按 OR
- 不同字段之间按 AND

示例：

```text
time in [T1, T2]
AND source in {surfaceflinger, init, ivcd}
AND pid in {1234, 5678}
AND keyword in {"error", "fail"}
AND NOT exclude in {"chatty", "heartbeat"}
```

建议支持两类过滤：

- 通用过滤：`time/source/keyword/exclude/pid`
- 格式化过滤：`tag/app/ctx/proc/boot_session/level`

兼容性要求：

- 第一阶段必须支持把上述统一查询映射回现有 `logcat` 风格入口
- 新增能力不能破坏旧参数语义

### 8.11 Boot Session 检测设计

#### 8.11.1 总体思路

当前阶段先以 Android 日志为主做 boot session 检测，但 session 模型不能绑定 Android 专用字段。

建议结果状态：

- `confirmed`: 有明确起止证据
- `inferred`: 由高置信推断得到
- `partial`: 只看到了 boot 的一段

#### 8.11.2 Android session 起点

Android 起点证据建议采用“多证据加权”，而不是依赖单条日志。

高优先级起点证据：

- `init` 早期启动日志
- `boot_progress_start`
- `SystemServer` 启动阶段日志
- 明确的 `reboot reason` / `boot reason`
- 时间倒退或日志连续性中断后出现新的系统早期事件

#### 8.11.3 Android session 终点

终点不一定等于关机完成，离线日志里通常只能识别“该 boot 的可观察结束”。

建议终点来源：

- 下一次 boot 的起点前一刻
- 当前日志输入的末尾
- 明确的 shutdown / reboot 事件

#### 8.11.4 Session 切分算法

建议算法：

1. 先解析全部 Android 事件
2. 按时间排序
3. 识别起点候选和终点候选
4. 结合时间断裂、重启原因、系统早期事件做聚类
5. 生成 `BootSession`
6. 把同时间窗口内的其他格式事件再映射回对应 session

### 8.12 Android Boot 与 `dmesg` 关联设计

#### 8.12.1 结论先行

不要被动从现有自然日志里“找一个唯一事件”做长期主关联。最稳的方案是主动定义一条 boot correlation event。

#### 8.12.2 推荐方案：boot correlation event

建议在采集链路里引入统一关联事件，内容至少包括：

- `boot_id`
- `boot_reason`
- `slot`
- `kernel_release`
- `build_fingerprint` 或等价版本信息
- 采集时间

建议同时在以下位置记录：

- Android 早期日志
- `dmesg` 导出文件头或 metadata
- 打包清单文件

这样关联逻辑可以简化为：

1. 优先按 `boot_id` 精确匹配
2. 没有 `boot_id` 时再按其他字段组合降级

#### 8.12.3 如果当前无法改采集链路

如果现阶段只能处理历史包，建议关联优先级如下：

1. `boot_id`
2. `boot_reason + slot + kernel_release`
3. Android boot 起点时间窗口与 `dmesg` 采集时间窗口
4. 同一 boot 的早期系统事件指纹

这里的“事件指纹”建议由多项组成，而不是单条日志，例如：

- 早期 `init` 序列
- `ro.boot.bootreason` 或等价日志
- 内核版本
- SELinux / AVB / dm-verity 相关早期事件

#### 8.12.4 如果必须只选一个最强事件

如果必须在现有日志里选一个“关联性很强的事件”，我的建议是：

- 优先选 `boot reason` 相关事件，并与 boot 起始时间窗口组合使用

原因：

- 它比业务日志更系统级
- 它比 `boot_completed` 更接近 boot 起点
- 它在 Android 和内核启动链路里都更容易留下痕迹

但要明确：

- 单独一个 `boot reason` 仍不足以作为长期唯一键
- 最终还是应该升级为 `boot_id` 或 boot correlation event

### 8.13 `dmesg` 建模建议

`dmesg` 至少要支持两种形态：

- 文本 `dmesg`
- 从其他来源导出的 kernel log

建议字段：

- `mono_time`
- `wall_time`，若采集端已补齐
- `kernel_seq`
- `subsystem`
- `cpu`
- `level`
- `message`
- `boot_id`

关键约束：

- `dmesg` 的相对时间不能直接和 Android 绝对时间混排
- 只有当某次 boot 已建立绝对起点，才允许把 `mono_time` 投影成绝对时间
- 这种投影必须被标记为 derived，而不是 exact

### 8.14 输出与可解释性设计

工具输出不能只给“命中结果”，还要能说明这些结果为什么属于某个 boot。

建议支持两类输出：

- 人类可读文本输出
- 机器可读 JSON 输出

Boot session 和 `dmesg` 关联结果建议带 evidence，例如：

```json
{
  "boot_session_id": "boot-003",
  "confidence": "inferred",
  "matched_dmesg": "dmesg_03.txt",
  "evidence": [
    "boot_reason=hard_reset",
    "kernel_release=5.15.x",
    "start_time_delta=2.1s"
  ]
}
```

### 8.16 分阶段实施建议

#### 阶段 1

- 以“可替换现有 `logcat` 脚本”为验收门槛
- 保留 `logcat` 命令名和现有主要参数
- Python CLI 骨架
- 输入层支持文件、目录
- parser 至少支持 Android，并对 DLT 文本、journal 文本、Unity 提供同等查询入口
- 统一事件模型
- 基础时间模型
- 兼容查询：`-f/-p/-s/-k/-e/-t/--plain`
- 基础查询：`time/source/keyword/exclude/pid`
- 兼容输出：文件名、行号、纯文本模式
- Android boot session 检测第一版

#### 阶段 2

- 输入层支持 `.gz/.tgz`
- `dmesg` parser
- `dmesg` 与 Android boot 关联
- JSON 输出
- session evidence 输出

#### 阶段 3

- 二进制 DLT 解码
- boot correlation event 落地
- `boot_id` 精确关联
- 性能优化与缓存

### 8.17 当前推荐结论

当前最合理的设计路线是：

1. 用 Python 重构为多阶段日志分析框架
2. 第一阶段先达到现有 `logcat` 的功能 parity
3. 在 parity 基础上统一事件模型和时间模型
4. 先把 Android、DLT 文本、journal、Unity 跑通
5. boot session 当前以 Android 为主切分
6. `dmesg` 关联当前先做启发式，长期改为 `boot_id` 或 boot correlation event

也就是说，短期目标不是直接追新功能，而是先让新工具完整接住现有 `logcat` 的使用场景；在此基础上，再逐步扩展到多格式时间查询、boot session 检测和 `dmesg` 关联。
