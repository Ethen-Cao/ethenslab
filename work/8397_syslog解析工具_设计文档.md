## 1. 背景与目标

开发人员拿到一份 syslog(若干滚动日志文件)后,排查问题前需要先建立"时间轴认知":

- 系统经历了 **几次冷启动**,每次冷启动 **落在哪些日志文件**、**持续多久**;
- 系统 **何时进入 STR(Suspend-To-RAM)**、**何时从 STR 复归**;
- 在 **系统运行期间**(两次冷启动之间、STR 复归到下次 STR 进入之间)发生了哪些 **异常事件**;
- 最终把这些信息汇总成一份 **HTML 报告**,便于快速定位问题与对应日志位置。

本工具即自动化上述过程。**异常事件的定义与检测规则会持续扩展**,因此检测部分必须做成可插拔的规则引擎。

## 2. 范围

### 2.1 本期目标(In Scope)
1. 加载 / 排序一批 syslog 文件(处理命名规律、50MB 滚动、跨重启序号连续)。
2. 解析单行日志为结构化记录。
3. 检测 **冷启动**:次数、每次起止、覆盖文件、持续时间。
4. 检测 **STR 进入 / 复归**:时间点、配对成图(进入↔复归)。🔗依赖样例
5. 构建 **运行时间窗**(awake 窗口),并在窗口内 **检测异常事件**(可扩展规则引擎,先搭框架 + 少量内置规则)。
6. 生成 **HTML 报告**。

### 2.2 暂不在本期(Out of Scope,可后续迭代)
- 多域(PVM/GVM)日志的跨域时间对齐(⏳待确认是否需要)。
- 异常根因自动推断 / 关联分析(本期只做"检测 + 归类 + 定位")。
- 实时 / 流式解析(本期为离线批处理)。

## 3. 术语

| 术语 | 含义 |
|------|------|
| 冷启动 Cold Start | 整机断电后重新上电,内核重新引导(出现 `Booting Linux` banner、uptime 归零) |
| STR | Suspend-To-RAM,挂起到内存(休眠)。STR 进入=挂起;STR 复归=唤醒恢复 |
| 启动会话 Boot Session | 一次冷启动到下一次冷启动之间的全部日志 |
| 运行时间窗 Awake Window | 系统处于"运行/唤醒"状态的时间段,异常检测的作用域。边界:冷启动就绪 / STR 复归(起) → STR 进入 / 下次冷启动(止) |
| uptime | 每行日志第 2 列,内核启动后的毫秒数(详见 §5.2) |
| PVM / GVM | 平台上的两个虚拟域(日志中存在 `pvm_uptime` / `gvm_uptime`)。本批样本为 PVM syslog |

## 5. 关键事实(已验证,作为设计基础)

> 完整验证过程见 **附录 B/C**。这里给出设计直接依赖的结论。

### 5.1 文件命名 & 排序
- 格式:`{序号}_{YYYYMMDD}_{HHMMSS}.log`,序号 **全局递增且跨重启连续**。
- 文件写满 **≈50MB(52428800 字节)** 即滚动;同名 `.zip` 为该日志压缩归档。
- **排序必须按序号(而非文件名时间戳)**,因为冷启动瞬间 RTC 未同步,时间戳不可信(见 §5.3)。
- 序号可能有缺号(未导出),按序号顺延即可。

### 5.2 单行格式与 uptime
```
2026-06-17 14:05:10.274  75392  [D]  qcxserver  1400  [tid:1601]: <消息>
└── 墙钟时间 ──────────┘  └ uptime ┘ └级别┘ └模块┘ └PID┘ └─── 正文 ───┘
```
- 第 2 列 = **uptime(毫秒,内核启动后)**。已验证:同一文件内 Δuptime 与 Δ墙钟一致,且与日志中 `pvm_uptime` 字段吻合。
- **uptime 是判断冷启动最稳的信号**(冷启动后从 ~1080ms 重新计数)。
- ⏳待确认:是否所有模块的行都带该列且语义一致(kernel 行 vs 应用行)。解析器需对缺列/异常行做容错。

### 5.3 冷启动检测(已验证)
三重佐证,任一为主、其余为辅:
1. **内核 banner(主)**:`Booting Linux on physical CPU`(出现在冷启动文件第 1 行);`Linux version ...` 第 2 行佐证。
2. **uptime 复位**:第 2 列回到 ~1080ms。
3. **墙钟跳变**:变为出厂默认 `2025-05-30 02:48:2x`(RTC 未同步)。

> ⚠️ `Boot Linux`(精确串)在日志中**不存在**,不能搜它;须搜 `Booting Linux`。

## 6. 总体架构与详细设计

### 6.0 处理流水线
```
syslog 目录
   │
   ▼
[1] Loader 加载器        —— 发现文件、按序号排序、(可选)解压 zip、流式读取
   │  产出:有序的文件流
   ▼
[2] LineParser 行解析     —— 行→结构化记录(墙钟/uptime/级别/模块/pid/tid/正文)
   │  产出:LogRecord 流
   ▼
[3] TimelineBuilder 时间轴 —— 冷启动检测、STR 进入/复归检测、切分启动会话 & 运行时间窗
   │  产出:Sessions / AwakeWindows / 电源事件
   ▼
[4] DetectorEngine 异常引擎 —— 可插拔规则,在运行时间窗内扫描日志,产出 AnomalyEvent
   │  产出:归类、去重、定位(文件:行)的异常列表
   ▼
[5] Reporter 报告器        —— 汇总为 HTML(+ 导出中间 JSON)
   │
   ▼
report.html  +  result.json
```

### 6.1 Loader 加载器
- 扫描目录,正则提取序号,**按序号升序**排列。
- 默认读 `.log`;若某序号只有 `.zip`(对应 `.log` 已被清理),则从 zip 读取。🔗依赖/⏳待确认:zip 内是否就是同名 log 的完整内容(需抽样核对)。
- **流式逐行读取**(单文件 50MB、整批可达 ~22GB,严禁全量载入内存)。
- 记录每条记录来源 `(文件名, 行号, 全局序号)`,供报告定位。
- 性能:支持按需只扫描"指定会话/时间窗"涉及的文件(报告生成时的下钻可懒加载)。

### 6.2 LineParser 行解析
- 用正则把行拆为 `LogRecord`:
  ```
  LogRecord = { wall_time, uptime_ms, level, module, pid, tid, message,
                source_file, line_no, global_index }
  ```
- 容错:
  - **多行消息**(无前缀的续行)归并到上一条记录。
  - 缺列 / 格式异常行:保留原文,标记 `malformed=true`,不中断。
  - 墙钟时间标记 `clock_trusted`(false 当处于冷启动初期 RTC 未同步窗口)。

### 6.3 TimelineBuilder 时间轴构建

#### 6.3.1 冷启动检测(已验证,见 §5.3)
- 主规则:行匹配 `Booting Linux on physical CPU`。
- 每个匹配 = 一次冷启动锚点,记录 `(序号, 文件, 行号, uptime起点)`。
- **会话切分**:按序号排序后,从一个冷启动锚点(含)到下一个锚点之前为一个 **启动会话**。
- 首个锚点之前的文件归入 **"会话 0(boot 未捕获)"**(本批样本中文件 419 起点 uptime 已是 75s,见附录 C)。

#### 6.3.2 冷启动"持续时间"
- ⏳待确认定义,候选(设计上做成可配置):
  - (a) **开机耗时**:从 `Booting Linux` 到 "系统就绪" 标志(开机时间 KPI)。**需要"就绪/首帧/可用"标志** 🔗依赖样例。
  - (b) **会话时长**:该次冷启动的 uptime 跨度(到下次冷启动 / 日志结束)。可由 uptime 直接算,无需额外标志。
- **当前默认**:先实现 (b)(无依赖、确定可算);(a) 待"就绪标志"确认后补充。报告中两者可并列。

#### 6.3.3 STR 进入 / 复归检测
- 🔗**依赖:STR 进入 / 复归的日志样例与关键字由用户后续提供。** 当前留出接口与数据结构,规则置空。
- 设计预留:
  - `STR_ENTER` 规则:匹配"进入挂起"特征行 → 记录时间点(墙钟 + uptime)。
  - `STR_RESUME` 规则:匹配"复归/唤醒完成"特征行 → 记录时间点。
  - **配对**:同一会话内按时间顺序将 ENTER↔RESUME 配对,计算每次 STR 时长;处理孤立事件(只进未归 / 只归未进,可能跨文件丢失)。
  - 注意:STR 复归 **不会** 重置 uptime(区别于冷启动),这是区分"复归"与"冷启动"的依据。⏳待确认是否存在不重置 uptime 的"热重启"需单列。

#### 6.3.4 运行时间窗(Awake Window)
- 由冷启动锚点 + STR 事件切分:
  - 窗口起点 = 冷启动就绪 **或** STR 复归;
  - 窗口止点 = STR 进入 **或** 下次冷启动 **或** 日志结束。
- 异常检测仅在 Awake Window 内进行;STR 挂起期间默认不算异常作用域(⏳待确认是否需要检测挂起期/唤醒源)。

### 6.4 DetectorEngine 异常事件检测引擎(可扩展)

> 核心诉求:**异常定义与规则会持续扩展**。因此规则与代码解耦,新增规则不改主流程。

- **规则定义(声明式,建议 YAML/JSON 规则文件 + 少量复杂规则用 Python 插件):**
  ```yaml
  - id: kernel_oops
    name: "Kernel Oops/Panic"
    category: kernel
    severity: critical
    match:                 # 任一命中
      - regex: "Kernel panic"
      - regex: "Unable to handle kernel"
      - regex: "BUG: "
    scope: awake           # awake | session | all
    dedup:                 # 去重,避免刷屏
      window_ms: 5000
      key: "module+message_signature"
    context_lines: 20      # 报告中附带的上下文行数
    enabled: true
  ```
- **规则类型**(预留):单行正则、多行/状态机(如"X 后 N 秒内未见 Y")、阈值(单位时间内 ERROR 计数突增)、模块特定解析器。
- **内置示例规则(初始集,待与用户对齐后扩充):**
  - `kernel_panic` / `oops` / `BUG:`(critical)
  - `*** *** ANR / watchdog / hung task`(待确认平台关键字)
  - 高频 `[E]` 错误聚类(按 模块+消息签名 聚合计数)
  - 进程崩溃 / `signal` / `tombstone`(待确认关键字)
  - **boot loop 检测**:相邻冷启动间隔过短(本批样本 791→792 即疑似,见附录 C)
  > 🔗依赖:具体异常清单、关键字、严重级别由用户分批提供;每条规则附"判定依据 + 样例行"。
- **输出 AnomalyEvent:**
  ```
  AnomalyEvent = { rule_id, category, severity, first_time, count,
                   sample_message, source_file, line_no,
                   session_id, awake_window_id }
  ```

### 6.5 Reporter HTML 报告

- **单文件、自包含 HTML**(内联 CSS/JS,便于分发;⏳待确认是否需要图表库/可离线)。
- 建议结构:
  1. **概览**:冷启动次数、STR 次数、异常总数(按严重级别)、日志时间跨度、文件数/大小。
  2. **时间轴**:冷启动 / STR 进入 / STR 复归 / 异常 在一条时间轴上(可用 uptime 或墙钟,标注时间不可信段)。
  3. **启动会话表**:每次冷启动 → 覆盖文件区间、持续时间、该会话异常数(可下钻)。
  4. **STR 事件表**:进入 / 复归配对、每次时长。
  5. **异常列表**:按严重级别/类别筛选,展示 `文件:行 + 上下文`,可复制定位。
  6. **异常规则附录**:本次启用了哪些规则及版本,便于复现。
- 交互:筛选、折叠、按会话/时间窗分组;点击异常跳转到来源 `文件:行`。

### 6.6 中间结果 JSON(result.json)
- 时间轴、会话、STR、异常的结构化导出,作为 HTML 的数据源 + 回归测试基线 + 二次分析入口。
