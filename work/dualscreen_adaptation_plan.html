二、当前架构的问题根因

2.1 架构缺陷全景

2.2 getFrameRateDivisor 的 90/60 bug

这是方案二拟修复的核心 bug。源码（RefreshRateSelector.cpp:1791–1805）：

numPeriods = 90.0 / 60.0 = 1.5
numPeriodsRounded = std::round(1.5) = 2.0
abs(1.5 − 2.0) = 0.5 > 0.0009 (kThreshold) → return 0

→ isVSyncInPhase: divisor (0) ≤ 1 → 始终返回 true → 无节流

2.3 TPT 重复机制（默认行为：follower_display_backpressure=OFF）

关键洞察：默认路径（follower_display_backpressure=OFF）中，SF 不会"跳过"副屏——它每帧都执行 commit。 问题是两个连续的 SF 心跳可能指向 相同的副屏 VSync 边界（均为 33.3ms，均为 50.0ms）。 第二个 commit 覆盖 HWC 队列中第一个 commit 的帧。只有 一半的 commit 帧能真正上屏（按 HWC 的 60Hz 节拍），被覆盖的帧则是无效工作。

三、方案一：彻底解耦机制（上策）

目标：主屏和副屏以各自独立的 VSync 节奏完全自主运行，架构上互不干扰。
适用对象：具备深度 AOSP 定制能力的车机/多屏设备团队。
效果：每个 App 接收到所附着显示器对应的 VSync；每个显示器独立控制 commit/composite 时序。

3.3 方案一的可行性分析

✓ 可行部分
  • AOSP 已具备完整的基础设施：每显示器独立的 VsyncSchedule（含 Reactor+Predictor+TimerQueue）、独立的 RefreshRateSelector、独立的 HWC present
  • 源码中的 TODO (EventThread.h:267) 表明 per-display EventThread 是已知的计划方向，架构预留了扩展点
  • AIDL 接口扩展（新增 displayId 参数）是 Android 框架层改造的标准模式，版本兼容可管理
  • CompositionEngine 已经以 displayId 为 key 进行分片，天然支持按显示器操作
✗ 风险与不确定性
  • (严重) 高通 HWC HAL 并发能力未知：闭源 HAL 的 validate/present 方法是否支持两个显示器并发调用？如果 HAL 内部有全局互斥锁，per-display 合成仍会被串行化——等于白改
  • (严重) GPU 驱动多队列支持：两个显示器并发走 GPU 合成路径时需要 GPU 驱动支持多 command queue；高通 Adreno 驱动需确认
  • (中等) SF 唤醒频率：单个 SF 线程同时服务两个独立的 VSync 回调时，Looper 唤醒频率可能从 90Hz 变为 ~150Hz（90+60 事件交叉），需确认功耗是否可接受
  • (中等) App 框架改造面广：Choreographer、ViewRootImpl、DisplayEventReceiver 的改造涉及 Java 框架层，需全面的回归测试（尤其影响 WindowManager、Insets、Popups 等跨显示器场景）
  • (低) 跨显示器 Surface 迁移：当 Layer 从一个显示器拖拽到另一个显示器时，Choreographer 的 VSync 连接需要切换——这是边缘情况但必须处理

四、方案二：基于 AOSP 原生逻辑的算法改良（中策）

目标：在 Pacesetter 统一心跳架构内，修复 App 侧节流 bug 并启用 Follower 背压保护。
适用对象：改动窗口有限，无法进行大规模架构改造的项目。
效果：App VSync 从"完全无节流"改善为"交替间隔（11.1/22.2ms）"；副屏通过背压机制减少无效 commit。

4.1 改造一：修复 getFrameRateDivisor — App 侧节流

Bug 根因

RefreshRateSelector::getFrameRateDivisor(90_Hz, 60_Hz)
→ numPeriods = 90/60 = 1.5
→ std::round(1.5) = 2.0 (C++ rounds halfway away from zero)
→ abs(1.5 − 2.0) = 0.5 > 0.0009 → return 0

下游 isVSyncInPhase：if (divisor == 0) return true → 全部放行

涉及的源码改动

文件改动内容
RefreshRateSelector.cpp:getFrameRateDivisor 返回值 0 改为 -1；或新增 getFrameRatePhaseDivisor 单独处理非整数倍情况
VSyncPredictor.cpp:isVSyncInPhase 当 divisor 为 -1 时，走 Bresenham 相位累加器分支而非直接返回 true
EventThread.h（或新增 per-connection 状态） 每个 EventThreadConnection 维护 phaseAccumulator 和 appFrameRate

4.2 改造二：强制启用 follower_display_backpressure

工作原理（Scheduler.cpp:437–468）

当 follower_display_backpressure = true 时，Follower 的参与条件由无条件 commit 变为条件性跳过：

pendingFenceCount = targeter.countPresentFencesPendingAt(frameBeginTime);
nextFollowerVsync  = display.schedulePtr->vsyncDeadlineAfter(frameBeginTime);

if (pendingFenceCount > 0 &&
    !(pendingFenceCount == 1 && nextFollowerVsync < expectedVsyncTime)) {
    continue;  // 跳过此 Follower 在此心跳中的 commit
}

关键约束：跳帧不是固定节拍（非"每 3 跳 1"），而是取决于 Present Fence 的实际 signal 时机。在高通 HWC 的具体硬件行为上，模式会趋于

规律，但无法从源码中给出保证。

典型 90Hz+60Hz 下 fence 时序可能的模式：

心跳pendingFenceCountnextFollowerVsync < expectedVsync?决策
1 (T≈0)0（初帧无先例）—（跳过低条件为 pendingFenceCount=0）参与
2 (T≈11.1)1（16.6ms 的 fence 仍在 pending）16.6 < 22.2 → 是参与（容忍单个）
3 (T≈22.2)1 或 2（取决于 33.3ms 的 fence）取决于 fence可能跳过

4.3 方案二的可行性分析

✓ 可行部分
  • Bresenham 累加器在数学上是可靠的——对 90/60 比完美产生 [发,发,拦] 模式
  • 改动范围小（~3 个文件），回归风险可控
  • follower_display_backpressure flag 已存在且经过单元测试（SchedulerTest.cpp 有覆盖），仅需在 .aconfig 中默认启用
  • 不依赖高通 HAL 的任何并发支持——不改变 compositor 调用模式
✗ 局限性与残余风险
  • (严重) App VSync 仍然不均匀：11.1ms/22.2ms 交替间隔，App 在 11.1ms 短间隔内需要比均匀 16.67ms 更快的渲染时间。如果 App 渲染时间接近 16.6ms，短间隔必丢帧
  • (严重) SF 仍然做无效工作：Follower 在参与帧中仍可能产生 TPT 重复——backpressure 只减少 commit 次数，不能消除 TPT 对齐问题
  • (中等) 跳帧模式依赖硬件时序：实际跳帧率不仅取决于 90/60 的数学关系，还取决于 HWC 的 fence signal 延迟——两台不同的 8397 设备可能表现不同
  • (中等) 功耗改善不完整：SF 仍以 90Hz 周期性被唤醒，只是偶尔跳过副屏的 commit——主唤醒频率未改变
  • (低) 优化残余：Bresenham 累加器需要 per-connection 状态维护；需要考虑连接数量增加时的内存开销

五、方案三：频率规避方案（下策）

目标：完全避免非整数刷新率比的场景。当检测到副屏激活时，将主屏刷新率从 90Hz 切换为 60Hz 或 120Hz——二者均为 60Hz 的整数倍。
适用对象：无法改动 AOSP 渲染管线的快速适配。
效果：App 侧节流完全正确；TPT 重复仍然存在但更规律；牺牲 90Hz 主屏体验。

5.3 方案三的可行性分析

✓ 可行部分
  • 无需改动渲染管线（VSync / EventThread / Scheduler / Compositor 均保持原生）
  • App 侧节流完美工作——整数比下 getFrameRateDivisor 返回正确值
  • 实施成本最低——仅涉及 DisplayMode 策略层代码
✗ 核心牺牲与残余风险
  • (严重) 主屏体验降级：切换到 120Hz 意味着 90Hz 从此不可用——这是用户可感知的主屏流畅度退化（如视频匹配 90fps 内容的场景）；如果降为 60Hz，则主屏完全失去高刷优势
  • (严重) TPT 重复问题未解决：即使整数比下，副屏仍有约 50% 的 commit 帧因 TPT 重复被静默覆盖。功耗浪费未改善
  • (中等) 动态切换延迟：副屏冷插拔、副屏上启动新 App 需要触发刷新率切换 + 等待新模式稳定——过渡期间可能出现闪烁
  • (中等) 120Hz 不一定是可用模式：需确认 8397 平台的面板规格是否支持 120Hz——如果不支持，则唯一选择是降至 60Hz
  • (低) 双屏中间状态：副屏插入/拔出时的切换时序需要精细管理，避免两个 Mode Change 同时进行导致冲突

七、全局风险矩阵与外部依赖

7.1 技术风险清单

  • P0 高通 HWC HAL 并发能力
    影响的方案：仅方案一
    风险：HAL 的 validate/present 实现可能存在全局互斥锁。若 HAL 串行处理两显示器，per-display 合成改造无济于事——两个显示器仍将被串行化。
    验证方式：联系高通获取 HAL 的线程安全文档；或编写测试代码从两个线程同时调用 validate/present，测量是否串行等待。
  • P0 GPU 驱动多队列支持
    影响的方案：方案一、部分影响方案二（backpressure 路径需更多 GPU 合成）
    风险：两个显示器并发走 GPU 合成时，Adreno GPU 驱动需支持并发 command submission。不支持则会导致 GPU 合成线程阻塞。
    验证方式：高通文档 + 压力测试（两显示器同时渲染不同内容）
  • P0 120Hz 面板模式可用性
    影响的方案：仅方案三
    风险：8397 平台的主屏面板可能不支持 120Hz 模式。如果不支持，方案三只能降到 60Hz。
    验证方式：检查 /sys/class/drm/ 或 HWC 支持的显示器模式列表
  • P0 副屏 TPT 重复问题在方案二/三中仍未解决
    影响的方案：方案二、方案三
    风险：无论 90Hz 还是 120Hz，TPT 重复率约 50%。HWC 接收 2 帧但只能显示 1 帧。用户仍能感知到微观卡顿（尤其是在副屏高速滚动场景）。
    缓解：方案一的 per-display 合成调度是唯一能根本消除 TPT 重复的方式。
  • P1 App 框架回归测试面广
    影响的方案：仅方案一
    风险：改动 Choreographer / ViewRootImpl 影响所有 App 的帧调度。跨显示器 Surface 迁移、Insets、Popups 场景需要全量测试。
    缓解：分阶段灰度发布；先在副屏独占 App 上验证
  • P1 Bresenham 累加器状态管理
    影响的方案：仅方案二
    风险：每个 EventThreadConnection 需维护 phase accumulator。大量连接时的内存开销和跨线程同步需考虑。
    缓解：使用 per-connection atomic accumulator；仅在 isVSyncInPhase 路径中初始化和更新
  • P2 FrameTimeline Jank 监控扩展
    影响的方案：所有方案
    风险：当前 FrameTimeline 仅用 Pacesetter 的 present fence 做 Jank 分类。副屏帧的 Jank 独立追踪需要扩展 per-display。
    缓解：方案一中可同步扩展；方案二/三可后续独立添加

八、推荐路径

分阶段渐进策略

Phase 0 — 必须首先完成的验证

  1. 高通 HWC HAL 并发测试：编写简易测试程序，从两个线程同时调用 IComposer::validate/present（不同显示器），测量是否有串行化延迟
  2. 120Hz 面板模式确认：检查 dumpsys SurfaceFlinger 中支持的显示模式列表，确认是否存在 120Hz 模式
  3. GPU 合成压力测试：两显示器同时播放视频，监测 GPU 利用率、合成延迟、是否有 fence timeout

Phase 1 — 快速改善（无论 Phase 0 的结果如何，均可推进）

  1. 修复 getFrameRateDivisor：对 90/60 等非整数比引入 Bresenham 相位累加器
  2. 在 .aconfig 中将 follower_display_backpressure 默认值为 true
  3. 添加 systrace 埋点：监测 Follower TPT 重复率、App VSync 间隔分布

Phase 2 — 如需根本解决则推进至方案一

仅当 Phase 0 确认了 HAL+GPU 并发能力，且产品对副屏体验要求极高时，才推进至此阶段。 这是目前多屏车机领域领先方案的做法——主流 OEM 大多已完成类似的 per-display 改造。

若不推进至方案一，采用方案三作为后备

若 Phase 0 发现 HAL 无法并发，且 Phase 1 的改善后副屏体验仍不达标， 则启用方案三：动态将主屏从 90Hz 切换为 60Hz 或 120Hz（取决于面板能力）。 实施时需注意切换过渡期间的 Display Mode 稳定性。
